## Overview

This guide provides step-by-step instructions for deploying the Budget Management System to Render.com, a cloud hosting platform. The application is containerized using Docker and uses PostgreSQL as the production database.

### Application Stack

- **Framework:** Django 5.2.8
- **Database:** PostgreSQL (via Render)
- **Containerization:** Docker
- **Web Server:** Gunicorn
- **Static Files:** WhiteNoise
- **Platform:** Render.com

### Deployment Architecture

```
GitHub Repository
    ↓
Render Web Service (Docker)
    ↓
PostgreSQL Database (Render)
    ↓
HTTPS Endpoint (Render)
```

## Database Configuration

### Step 1: Create PostgreSQL Database

1. In Render Dashboard, click **"+ New"** (top right)
2. Select **"PostgreSQL"**
3. Fill in the database configuration:

   **Database Configuration:**
   ```
   Name: budget-db
   Database: budgetdb
   User: (auto-generated by Render)
   Region: Choose closest to your location
     - Oregon (US West) - Recommended for US
     - Frankfurt (EU) - Recommended for Europe
     - Singapore (Asia Pacific) - Recommended for Asia
   PostgreSQL Version: 15 (or latest available)
   Plan: Free
   ```

4. Click **"Create Database"**
5. Wait 1-2 minutes for provisioning

### Step 2: Note Database Connection Details

After creation, Render will display:

- **Internal Database URL** (for use within Render)
- **External Database URL** (for external connections)
- **Host, Port, Database Name, Username, Password**

**Important:** Render automatically provides the `DATABASE_URL` environment variable when you link the database to your web service. You typically don't need to set this manually.

### Step 3: Database Security Notes

- **Free Tier Limitations:**
  - Database spins down after 90 days of inactivity
  - 1 GB storage limit
  - Automatic backups (data deleted after 90 days of inactivity)
  
- **Production Recommendations:**
  - Consider upgrading to paid plan for production use
  - Set up regular data exports
  - Monitor database usage

## Web Service Deployment

### Step 1: Create Web Service

1. In Render Dashboard, click **"+ New"**
2. Select **"Web Service"**
3. Connect your repository:
   - If not connected, click **"Connect GitHub"**
   - Select your repository: `budget-app` (or your repository name)
   - Click **"Connect"**

### Step 2: Configure Web Service

Fill in the service configuration:

**Basic Settings:**
```
Name: budget-app
Environment: Docker
Region: Same as your database (recommended)
Branch: main
Root Directory: (leave empty, or use "." if needed)
Dockerfile Path: Dockerfile (auto-detected)
Docker Context: (leave empty)
```

**Build & Deploy Settings:**
```
Build Command: (leave empty - Dockerfile handles this)
Start Command: (leave empty - Dockerfile handles this)
```

**Advanced Settings (Optional):**
```
Health Check Path: /
Auto-Deploy: Yes (deploys on every push to main branch)
```

### Step 3: Link Database

Before clicking "Create Web Service":

1. Scroll down to **"Add Database"** or **"Link Resource"** section
2. Select your PostgreSQL database: `budget-db`
3. This automatically links the database and sets `DATABASE_URL`

### Step 4: Create Web Service

1. Review all settings
2. Click **"Create Web Service"**
3. Render will:
   - Clone your repository
   - Build the Docker image
   - Start the deployment process
   - Provide a URL (e.g., `https://budget-app-abc123.onrender.com`)

**Note:** Initial deployment takes 5-10 minutes.

## Environment Variables Configuration

### Step 1: Access Environment Variables

1. In Render Dashboard, navigate to your **Web Service**
2. Click on the **"Environment"** tab
3. You should see `DATABASE_URL` already set (if database is linked)

### Step 2: Configure Required Environment Variables

Add the following environment variables one by one:

#### Variable 1: SECRET_KEY

**Purpose:** Django secret key for cryptographic signing and security.

**Configuration:**
```
Key: SECRET_KEY
Value: -9+p_m1%1qe5^9nbm33egcys)rfyaa$t%9#l#$8#ourcw0qx0b
```

**⚠️ SECURITY WARNING:**
- Never commit this value to Git
- Never share this key publicly
- Generate a new key for each deployment
- If compromised, generate a new key immediately

**How to Generate New Key:**
```bash
# Using Docker
docker-compose run --rm web python manage.py shell -c "from django.core.management.utils import get_random_secret_key; print(get_random_secret_key())"

# Using Python
python3 -c "from django.core.management.utils import get_random_secret_key; print(get_random_secret_key())"
```

#### Variable 2: DEBUG

**Purpose:** Controls Django debug mode. Must be False in production.

**Configuration:**
```
Key: DEBUG
Value: False
```

**Important:**
- Setting `DEBUG=True` in production is a security risk
- It exposes sensitive error information
- Only use `True` temporarily for debugging

#### Variable 3: ALLOWED_HOSTS

**Purpose:** List of host/domain names that Django can serve.

**Configuration:**
```
Key: ALLOWED_HOSTS
Value: your-app-name.onrender.com
```

**Example:**
If your Render URL is `https://budget-app-abc123.onrender.com`, set:
```
ALLOWED_HOSTS=budget-app-abc123.onrender.com
```

**Important:**
- Use only the domain name (no `https://` or trailing `/`)
- For multiple domains, use comma-separated: `domain1.com,domain2.com`
- Must match your actual Render URL exactly

**Finding Your Render URL:**
1. Go to your Web Service in Render Dashboard
2. Look at the top of the page - your URL is displayed
3. Or check the "Settings" tab → "Service URL"

#### Variable 4: CSRF_TRUSTED_ORIGINS (Optional but Recommended)

**Purpose:** Origins that are trusted for CSRF protection when using HTTPS.

**Configuration:**
```
Key: CSRF_TRUSTED_ORIGINS
Value: https://your-app-name.onrender.com
```

**Example:**
If your Render URL is `https://budget-app-abc123.onrender.com`, set:
```
CSRF_TRUSTED_ORIGINS=https://budget-app-abc123.onrender.com
```

**Note:** The application code automatically derives this from `ALLOWED_HOSTS` if not set, but explicitly setting it is recommended.

### Step 3: Complete Environment Variables Summary

**Final Environment Variables Checklist:**

| Variable | Value | Required | Notes |
|----------|-------|----------|-------|
| `SECRET_KEY` | `-9+p_m1%1qe5^9nbm33egcys)rfyaa$t%9#l#$8#ourcw0qx0b` | ✅ Yes | Generate new for each deployment |
| `DEBUG` | `False` | ✅ Yes | Must be False in production |
| `ALLOWED_HOSTS` | `your-app-name.onrender.com` | ✅ Yes | Match your Render URL exactly |

1. After adding each variable, click **"Save Changes"**

### Step 2: Create Superuser Account

Create an administrative user account to access the Django admin and manage the application.

1. In the same **Shell** tab (or open a new shell session)
2. Run the superuser creation command:

```bash
python manage.py createsuperuser
```

3. Follow the interactive prompts:

```
Username: admin
Email address: admin@example.com
Password: ********
Password (again): ********
Superuser created successfully.
```

**Security Best Practices:**
- Use a strong password (minimum 12 characters, mix of letters, numbers, symbols)
- Use a unique username (not "admin")
- Store credentials securely
- Consider using a password manager

### Step 2a: Understanding User Types

**Important:** After deployment, you have two options for accessing the application:

#### Option 1: Use Superuser for Everything (Simplest)

The superuser account you just created can:
- ✅ Access Django Admin at `/admin/` (full system access)
- ✅ Use the regular application at `/` (like any user)

**To use:**
1. Go to `https://your-app.onrender.com/login/`
2. Log in with your superuser credentials
3. You can use the app normally, or access `/admin/` for administration

**Note:** This is fine for personal use or small deployments.

#### Option 2: Create Separate Accounts (Recommended for Production)

**Superuser Account (Admin Only):**
- Created via: `python manage.py createsuperuser`
- Username: `admin` (or similar)
- Purpose: System administration only
- Access: Django Admin (`/admin/`) and regular app (`/`)

**Regular User Account (Daily Use):**
- Created via: Registration form at `/register/`
- Username: Your preferred username
- Purpose: Daily application usage
- Access: Regular app (`/`) only (no admin access)

**To create a regular user:**
1. Go to `https://your-app.onrender.com/register/`
2. Fill in the registration form
3. A household will be automatically created for you
4. Base template categories will be applied on first login

**Benefits of Separate Accounts:**
- Better security (admin credentials not used daily)
- Clear separation of roles
- Better for multi-user scenarios
- Follows security best practices

**Recommendation:** For production, create a superuser for admin tasks and register a separate regular user account for daily use.

## Troubleshooting

### Issue 1: Bad Request (400 Error)

**Symptoms:**
- Browser shows "400 Bad Request"
- Cannot access the application

**Causes:**
- `ALLOWED_HOSTS` not set correctly
- `ALLOWED_HOSTS` doesn't match Render URL

**Solution:**
1. Verify your Render URL in the dashboard
2. Check `ALLOWED_HOSTS` environment variable
3. Ensure it matches your URL exactly (no `https://`, no trailing `/`)
4. Save and wait for redeployment

### Issue 2: Server Error (500 Error)

**Symptoms:**
- Browser shows "500 Internal Server Error"
- Application doesn't load

**Common Causes:**
- Database migrations not run
- Missing environment variables
- Database connection issues
- Code errors

**Solution:**
1. **Check Logs:**
   - Go to Logs tab
   - Look for specific error messages

2. **Run Migrations:**
   ```bash
   python manage.py migrate
   ```

3. **Verify Environment Variables:**
   - Check all required variables are set
   - Verify `SECRET_KEY` is set
   - Verify `DATABASE_URL` exists (auto-set by Render)

4. **Temporarily Enable DEBUG:**
   - Set `DEBUG=True` in environment variables
   - Save and redeploy
   - Visit the site to see detailed error
   - Fix the issue
   - Set `DEBUG=False` again

### Issue 3: CSRF Verification Failed

**Symptoms:**
- "CSRF verification failed. Request aborted."
- Forms don't submit
- Cannot log in

**Causes:**
- Missing `CSRF_TRUSTED_ORIGINS`
- Incorrect CSRF settings

**Solution:**
1. Add `CSRF_TRUSTED_ORIGINS` environment variable:
   ```
   Key: CSRF_TRUSTED_ORIGINS
   Value: https://your-app-name.onrender.com
   ```
2. Ensure it includes `https://` prefix
3. Save and wait for redeployment

**Note:** The application code includes automatic CSRF configuration, but explicitly setting this variable is recommended.

### Issue 4: Database Connection Errors

**Symptoms:**
- "could not connect to server"
- "relation does not exist"
- Database errors in logs

**Causes:**
- Database not linked to web service
- `DATABASE_URL` not set
- Database not running

**Solution:**
1. **Verify Database is Linked:**
   - Go to Web Service → Environment tab
   - Check that `DATABASE_URL` exists
   - If missing, link the database in service settings

2. **Check Database Status:**
   - Go to Database dashboard
   - Verify it's running (not paused)

3. **Test Connection:**
   ```bash
   python manage.py dbshell
   ```
   If this works, database connection is fine.

### Issue 5: Static Files Not Loading

**Symptoms:**
- CSS/styles not applied
- JavaScript not working
- 404 errors for static files

**Causes:**
- Static files not collected
- WhiteNoise not configured correctly

**Solution:**
1. **Check Build Logs:**
   - Look for `collectstatic` in build logs
   - Should see "Static files collected"

2. **Manually Collect Static Files:**
   ```bash
   python manage.py collectstatic --noinput
   ```

3. **Verify WhiteNoise:**
   - Check that `whitenoise` is in `requirements.txt`
   - Verify middleware is in `settings.py`

### Issue 6: Application Spins Down (Free Tier)

**Symptoms:**
- First request takes 30+ seconds
- Application appears slow initially

**Cause:**
- Render free tier spins down after 15 minutes of inactivity

**Solution:**
- This is expected behavior on free tier
- First request after spin-down takes ~30 seconds
- Subsequent requests are fast
- Consider upgrading to paid plan for always-on service

### Issue 7: Cannot Access Shell

**Symptoms:**
- Shell tab doesn't connect
- "Connection timeout" errors

**Solution:**
1. Wait 10-20 seconds for connection
2. Try refreshing the page
3. Check that the service is running (not paused)
4. Try again later if service is spinning up

## Security Considerations

### Environment Variables Security

1. **Never commit secrets to Git:**
   - Use `.gitignore` to exclude `.env` files
   - Never commit `SECRET_KEY` or passwords

2. **Rotate secrets regularly:**
   - Generate new `SECRET_KEY` periodically
   - Update database passwords regularly

3. **Use strong passwords:**
   - Minimum 12 characters
   - Mix of letters, numbers, symbols
   - Unique for each service

### Application Security

1. **Keep DEBUG=False in production:**
   - Never enable debug mode in production
   - Only use temporarily for troubleshooting

2. **Keep dependencies updated:**
   - Regularly update `requirements.txt`
   - Review security advisories

3. **Use HTTPS:**
   - Render provides HTTPS automatically
   - Never use HTTP in production

4. **Regular security audits:**
   - Review Django security releases
   - Update Django and dependencies
   - Review application logs for suspicious activity

### Database Security

1. **Limit database access:**
   - Only link database to necessary services
   - Use strong database passwords

2. **Regular backups:**
   - Export data regularly
   - Store backups securely

3. **Monitor database access:**
   - Review database logs
   - Check for unusual activity

### C. Common Error Messages and Solutions

| Error Message | Solution |
|--------------|----------|
| "Bad Request (400)" | Check `ALLOWED_HOSTS` matches Render URL |
| "Internal Server Error (500)" | Run migrations, check logs, verify environment variables |
| "CSRF verification failed" | Add `CSRF_TRUSTED_ORIGINS` environment variable |
| "could not connect to server" | Verify database is linked and running |
| "relation does not exist" | Run `python manage.py migrate` |
| "Static files not found" | Run `python manage.py collectstatic --noinput` |
